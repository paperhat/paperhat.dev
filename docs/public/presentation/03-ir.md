# IR (Intermediate Representation)

After parsing CDX into an AST, Paperhat transforms that structure into an
**Intermediate Representation (IR)**.

The IR is the first stage where the system begins to treat the data as
**meaningful and consistent**, rather than merely well-formed.

Compared to the AST, the IR is:
- more regular
- more explicit
- easier to reason about
- independent of how the data was originally written

This is still not rendering.
It is **preparing meaning** in a stable form that later stages can rely on.

---

## How to read this (non-technical)

This is still the same recipe — but now the system has:

* made everything explicit
* filled in obvious structure
* assigned stable identifiers
* computed simple derived values

Nothing new was *invented*.
Anything marked as derived is clearly labeled as such.

The IR exists so later steps don’t have to guess, reinterpret, or re-analyze
the original document.

---

## For technical readers

The IR is a **canonical, normalized representation** of the authored data.

### What changes from AST → IR

Compared to the AST, the IR:

* normalizes structure into predictable shapes
* resolves implicit groupings
* assigns stable identifiers
* records document order explicitly
* computes simple, deterministic derivations

The AST preserves *how* something was written.
The IR preserves *what it means structurally*.

---

### Identity and stability

Identifiers in IR are:

* deterministic
* content-derived or rule-derived
* stable across equivalent inputs

This guarantees that:

* references remain consistent
* downstream graphs are stable
* rendering and querying do not depend on authoring accidents

No runtime state is involved.

---

### Derived values

Some values in IR are derived rather than authored
(e.g. total time computed from preparation + cooking).

Derived values are:

* deterministic
* explicitly marked
* reproducible from source data

This allows later stages to rely on them without ambiguity
while preserving provenance.

---

### Why IR is distinct from RDF

IR is still **application-internal**.

It is optimized for:

* clarity
* determinism
* transformation
* validation

The next step converts IR into a **semantic graph** that can be stored,
queried, and validated independently of this application’s internals.

Keeping these stages separate avoids leaking application structure
into the semantic layer.

---

### Why this step matters

Without a canonical IR:

* every renderer would re-interpret structure
* identity would be inconsistent
* derivations would be duplicated
* reasoning would become implicit and fragile

The IR is the last place where the system is still “thinking in trees”.
The next step moves into graphs.
